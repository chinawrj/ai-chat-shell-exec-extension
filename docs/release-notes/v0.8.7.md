# AI Chat Shell Exec v0.8.7

This release fixes duplicate-command adjudication so the shell server is the final authority and only an execution with a server-controlled completion proof in the resolved tmux pane can suppress a later request.

## Changes

- Removed frontend command-history and semantic-command blocking. A newly rendered helper is forwarded even when its command text matches an earlier helper or follows a failed, cancelled, or unavailable-target attempt.
- Kept only request-level scan debouncing for the exact rendered helper block; historical `shell-output`, SPA navigation, virtualized message replacement, and shared containers cannot turn a new helper into a frontend duplicate.
- Moved final duplicate adjudication to the shell server after resolving the actual tmux pane, keyed by pane instance, command, and actual cwd.
- Limited duplicate eligibility to server attempts whose command was both sent and confirmed complete. Running, failed, timed-out, prompt-rejected, target-unavailable, and otherwise unconfirmed attempts remain retryable.
- Added immutable internal attempt identities so colliding browser `callKey` values cannot overwrite concurrent ledger state.
- Made pane identity fail open: if tmux does not provide the server/session instance fields needed for a safe fingerprint, the request executes instead of risking a false duplicate.
- Made frontend request tracking resilient to virtualized DOM-node recycling and recognizes rendered `language-shell-output` code containers even when Markdown fences are absent from `innerText`.
- Disabled execution dedup for generic board CLIs because a command can print text that looks like the prompt before it has finished; board commands now always fail open.
- Preserved Force run as an explicit bypass of completed-execution dedup.
- Updated `doctor.sh` to require server protocol 4, matching the extension and server.
