# AI Chat Shell Exec v0.9.8

This release fixes the M365 Copilot case where a completed helper result appeared in the composer but the automatic send never happened, while preserving the five send-actuator functions shared by v0.8.6 and v0.8.9 byte-for-byte.

## What changed

- The outer ownership guard now recognizes the exact transformation used by `https://m365.cloud.microsoft/chat`: M365 Lexical removes inserted line breaks and appends its hidden `U+200B U+200C` caret sentinel.
- That compatibility is post-write only. Any pre-existing user draft containing characters—including spaces, tabs, NBSP, or text exactly equal to the pending helper—remains a strict composer conflict and is neither adopted, overwritten, nor sent.
- M365 submission proof is restricted to a fresh exact `.fai-UserMessage[role="article"]` root. M365 also flattens extension-inserted structured deliveries on submission, so only known fenced `shell-output`/identity-bearing agent payloads may use a strict no-trim newline-flattened comparison inside the plugin-owned before/after lifecycle. Non-structured generic newline collisions, Copilot responses, generic articles, altered output, added prefixes/suffixes, and whitespace changes cannot produce a receipt.
- Inactive generic tool and side-panel textareas no longer masquerade as a second chat composer and prevent the original send actuator from starting. Lower-scored strong reply composers and actively edited ambiguous textareas still fail closed to protect user drafts.
- A preflight ownership veto now remains retryable and is reported as guard readiness; the panel no longer falsely says the original actuator finished when it never started.
- Clearing or changing the composer is never presentation proof by itself. Without a newly rendered matching user-message root, the canonical result stays pending and no local tombstone or `run-result-presented` receipt is emitted. With auto-send disabled, insertion likewise remains pending until exact manual submission is observed.
- Host-rendered fenced `shell-output` messages can prove exact submission even when the renderer adds a language label or expansion UI, while still requiring the complete code payload.
- Empty-first framework composer redraws may settle into one bounded exact-text handoff. Search fields and extension-panel inputs are excluded from composer ownership, every visible non-empty strong reply draft and every actively edited ambiguous textarea vetoes sending, and delayed host submission rendering is checked before treating a cleared composer as user cancellation.
- The one-per-lifecycle actuator generation is recorded synchronously after the final guard preflight, with no awaited storage gap before the first original actuator call.
- Chrome 137+ E2E now loads the unpacked extension through an unsafe-authorized null-delimited CDP pipe; Chrome 116-136 retain the legacy loader path.

## Unchanged guarantees

- The shell server remains the sole final duplicate authority.
- Only commands actually executed by the shell server can become duplicate history; Ctrl+C after execution remains executed history.
- The five send-actuator functions remain source-locked byte-for-byte to both v0.8.6 and v0.8.9 (the functions are identical in those releases).
- Backend execution and composer writing remain single-shot. User removal or replacement cancels delivery without reinsertion.
- Duplicate-control diagnostics remain local and never enter model input.
