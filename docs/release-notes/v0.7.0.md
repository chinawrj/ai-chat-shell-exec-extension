# AI Chat Shell Exec v0.7.0

Tmux AI teammate release.

## Highlights

- Register a tmux pane that is already running an AI, such as Claude Code, as a `tmux-ai` slave from the web master floating panel.
- Deliver master tasks into that tmux pane and require the slave to return results through the local reply CLI, not by scraping tmux output.
- Generate a short per-task reply script so the tmux AI only needs to write the reply body file and run `sh ...-reply.sh`.
- Include a project-level Claude Code skill at `.claude/skills/tmux-ai-slave-reply/SKILL.md` so Claude running from this checkout or release source archive learns the reply-file and short-script workflow.
- Correlate web slave replies with the original task through `reply-to`, and reject wrong master, wrong task, stale reply, and duplicate reply variants with actionable hints.
- Add `agent-task-status` diagnostics so long-running delegated work can be inspected as waiting for poll, waiting for tmux-ai reply, or replied waiting for master delivery.
- Improve master-side pending delivery text so cached replies explain what is waiting and what the user should do next.
- Add opt-in real Claude tmux slave E2E coverage for web master panel -> tmux Claude -> short reply script -> web master page.

## Upgrade Notes

1. Download and unzip `ai-chat-shell-exec-extension-v0.7.0-source.zip`.
2. Reload the unpacked extension from the matching `extension/` directory.
3. Stop any older foreground shell server.
4. Start the current server with `./scripts/start_shell_server.sh`.
5. Refresh enabled chat tabs so the content script version matches the background worker.
6. Use the popup or floating-panel `Check` action and confirm it reports server protocol `4`, helper `1`, and `ForAI` readiness.
7. For tmux-ai slaves, open Claude Code in a tmux pane, register that pane from the master panel, then use the generated reply file and short `sh ...-reply.sh` command shown in each task prompt.
