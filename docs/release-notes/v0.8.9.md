# AI Chat Shell Exec v0.8.9

This patch fixes new shell helpers submitted after refreshing a page while an earlier helper is still running in tmux.

## What changed

- Shell helper execution is now serialized per resolved tmux pane.
- A refreshed page can submit a new helper while the old page's command continues. The new command waits safely and starts as soon as that pane is free.
- Queue waiting does not consume the new command's shell state timeout.
- Commands targeting different tmux panes continue to run independently.
- Moving or renumbering a tmux window does not bypass the queue because serialization follows the immutable tmux server and pane ids; Terminal vision self-tests use the same queue.
- Before a queued runner is dispatched, the server revalidates its socket, tmux server instance, and pane id. If tmux was reset while it waited, the stale request fails safely and can be submitted again against the new pane.
- Completed shell output reports `queued: true` and `queuedMs` when a command waited for its pane.

Because one tmux pane can have only one foreground command, a command queued behind an indefinitely running process still needs that process to exit or be interrupted with Ctrl+C before it can start.

## Validation

- Added a real isolated-tmux regression that keeps one command running, moves its tmux window to change the pane address, submits a new-page helper with a shorter state timeout, verifies it does not time out or get typed into the busy pane, and confirms queued work executes after either natural exit or Ctrl+C releases the pane.
- Added a real Chromium extension E2E that starts a long helper, refreshes the page, submits a new helper, and verifies the queued result reaches the refreshed page.
- Added coverage proving a different agent tmux pane remains concurrent and shell output includes queue diagnostics.
- Added a real tmux-server replacement regression proving a queued request cannot cross into a newly created pane that reused the old pane id, followed by a successful explicit retry.
