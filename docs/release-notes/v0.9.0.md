# AI Chat Shell Exec v0.9.0

This release makes the shell server the final and sole duplicate authority, strengthens long-running tmux execution recovery, and prevents internal duplicate diagnostics from being sent to the model.

## Duplicate and result-delivery semantics

- A shell command is eligible for duplicate treatment only after the shell server proves that the command reached the resolved tmux pane and completed there. Browser scan state and browser ledgers are never execution authority.
- The authoritative key is bound to the immutable pane instance, the full command, and the actual cwd captured at the queue head. Failed, cancelled-before-execution, timed-out or unconfirmed requests remain retryable.
- The pane identity includes the current pane shell PID. `tmux respawn-pane` therefore starts with fresh execution history even when tmux reuses the same pane id; missing pane-shell identity disables deduplication.
- Every real execution receives a canonical `executionId`. A presentation receipt records when its result has actually reached the chat.
- A later duplicate whose result was already presented is shown only in the local extension panel. No duplicate error, reason, skipped flag, or other extension diagnostic is inserted into the composer for the model.
- If a page was refreshed before a completed result could be presented, the server can replay the bounded original result once as a clean recovered shell result. Duplicate/replay diagnostics are stripped before model delivery.
- Presentation receipts are persistent and monotonic across duplicate status recovery. A failed receipt is retried locally without reinserting the result, and same-execution pending deliveries are coalesced within the tab/page lifecycle.

## Long-running and interrupted commands

- Shell and Terminal-vision runners are serialized by immutable tmux server and pane identity, including across shell-server restart and page refresh.
- Persistent pane ownership prevents a refreshed page or restarted server from typing another runner into a still-busy pane. New work waits without consuming its own runner-state timeout.
- Ctrl+C after the executed marker returns promptly as exit 130 and counts as executed history. Ctrl+C before that marker also returns promptly, but remains retryable and cannot suppress a future run.
- Queue-head processing revalidates the pane instance and captures cwd only when execution can actually begin, preventing stale-pane or pre-queue duplicate claims.

## Recovery and related hardening

- Completed shell and board responses are cached before composer delivery. Same-page recovery retries local delivery only and never resends the underlying helper request.
- Runtime-channel loss uses read-only shell/board status recovery. A full reload relies on persistent ownership and backend adjudication.
- If ambiguous status recovery itself becomes unavailable, the failure remains local and that rendered helper is not reopened or resubmitted; a genuinely new rendered request may still ask the backend for a fresh adjudication.
- Board status is isolated from shell history and can never become shell duplicate authority.
- Identical agent-message retries are idempotent after response loss, and composer delivery now uses stricter ownership and stale-control checks.
- The local protocol is now version 6 and Chrome 116 is the minimum supported browser.

## Validation

- Added real tmux coverage for long-running commands, queueing, natural exit, Ctrl+C before and after execution, pane replacement, server restart, persistent ownership recovery, and duplicate replay bounds.
- Added real Chromium coverage for page refresh, over-30-second operations, clean recovery of an unpresented result, and local-only handling of an already-presented duplicate.
